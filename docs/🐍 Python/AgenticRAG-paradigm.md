# 什么是 Agentic RAG？

简短描述：Agentic RAG 就是在传统 RAG 的基础上，让模型具备“自主决策能力”，可以主动规划、选择工具、迭代查询，从而更智能地完成任务的一种 RAG 形式。传统 RAG（Retrieval-Augmented Generation）只是：“**提问 → 检索 → 回答**”，流程固定，被动等待输入。

**Agentic RAG 则让模型像一个小智能体（Agent）一样：**

- 会自己判断当前是否需要检索；
- 会把一个大问题拆成小步骤；
- 会在检索不够时自动继续查、继续验证；
- 会使用多个工具（检索、API、计算工具等）；
- 会将结果进行推理和整合；
- 会循环迭代直到完成目标。

**为什么谈 Agentic RAG？**

1. Retrieval‑Augmented Generation（RAG） 是当前很多基于大模型 + 外部知识库系统的基础技术：它通过向量检索 + 文本生成，让大模型能够结合外部知识、减少“凭记忆胡编”的问题；
2. 但传统 RAG 的 “检索 → 生成” 是线性的、单步的 —— 对于复杂任务（多跳推理、多工具调用、多文档综合）能力有限；
3. 因此，引入 Agentic RAG：通过把大模型封装为“智能体 (Agent)” — 它不仅能生成，还能“思考、规划、决策、检索、评估、迭代” —— 提升系统灵活性与智能水平。

# Agentic RAG 的七种范式

## 单个 Agent RAG

Single-Agent RAG 是 Agentic RAG 的最基础形式，系统中只引入一个代理执行任务，主要用于处理多数据源检索任务。

**架构特点**

- 核心角色：一个代理负责整个任务流程，包括检索🔍、工具选择⚙、上下文判断🧠和答案生成✍；
- 路由能力：该代理可以作为 Router，根据查询内容动态决定从哪个信息源或者工具中提取信息；
- 接入工具：支持接入多个外部工具或者数据源，如：
  - 向量数据库（如 Chroma）；
  - Web 搜索引擎；
  - 内部 API 等。

**应用场景：**

- FAQ 系统：根据问题类型判断是从数据库、网页还是 FAQ 文档中获取答案；
- 客服代理：接入多个业务系统（工单、邮件、用户行为日志等）提供支持。

**优缺点：**

- 优点：
  - 实现简单、结构清晰，适合快速验证想法或者搭建 MVP；
  - 适用于数据源不复杂、任务边界明确的场景；
  - 对比传统 RAG 更加智能灵活、可以动态选择信息源。
- 缺陷：
  - 所有任务集中在一个代理上，容易过载；
  - 代理功能复杂，不利于模块化优化；

## 多个 Agentic RAG

numti-agent RAG 是更加复杂的 Agentic RAG 实现，系统中引入多个功能划分明确的代理，通过主控代理（master agent）协调各个专职代理（specialized agents）协同完成任务。

> 多代理 RAG 是对单一代理 RAG 的自然进化，用团队协作换来了更强的处理能力。

**架构特点：**

- master agent：
  -  负责任务拆解🧩，子任务分配📋、汇总各个子任务结果📊。
- specialized agents：
  -  每个专职代理负责特定检索任务，如：
  - agent A：📂检索内部文档数据；
  - agent B：🔍基于 Web 进行实时搜索；
  - agent C：📧调用公司 API 查询邮箱。

**应用场景：**

- 企业级问答系统：结合私有知识库、ERP 系统数据、外部 API、公共网页搜索；
- 多模态信息检索：文本代理检索文本、图像代理调用图像识别模块、视频代理查找相关片段；

**优点**

- 模块化强：任务明确、职责清晰，便于扩展和优化；
- 支持复杂的、多阶段的任务链；
- 各个代理可以独立优化，便于并行处理，提高效率；

**缺点**

- 架构更加复杂，需要调度和通信机制；
- 代理间协调不当可能会造成冲突、冗余、或者信息损失；
- 成本更高（模型调用频率上升），需要优化任务拆解逻辑。

## 层级式 Agentic RAG

采用多层级代理架构（Hierarchical agents），由上层代理（如主控代理）协调下层专职代理（如检索代理、生成代理）完成复杂任务。

**架构特点**

- 主控代理：负责任务分解、子任务分配和结果整合；
- 专职代理：各自承担特定功能，如检索、生成、验证等；
- 层级协作：上层代理可以监督和指导下层代理的操作。

**适用场景**

需要多步骤推理和复杂任务管理的应用，如多文档问答、跨领域信息整合等。

**优势**

- 提高系统模块化和可扩展性；
- 增强任务处理的灵活性和准确性；

## 纠错型 Agentic RAG

在层级式 Agentic RAG 的基础上，引入反馈机制，代理在产生初步回答后进行自我评估，识别并纠正潜在错误、提升回答质量。

**架构特点**

- ✍初步生成：代理根据检索结果产生初步回答；
- 🧐自我评估：引入“批评者（critic）”模块，评估回答的准确性和一致性；
- 🔁迭代优化：根据评估结果，代理对回答进行纠正，直到满足质量标准；

**适用场景**

对回答准确性要求高的应用，如医疗咨询、法律回答等。

**优势**

- 显著降低错误率和幻觉；
- 提升用户对系统回答的信任度。

## 自适应 Agentic RAG

智能体根据任务需求和中间结果，动态调整检索策略和工具选择，实现更灵活的任务处理。

**架构特点**

- 🧭动态决策：代理可以根据当前的上下文，决定是否需要进一步检索或者更换工具；
- 🔧多工具集成：支持接入多种工具和数据源，如向量数据库💾、Web 搜索🌐、API调用📩等；
- 📈实时反馈：根据中间结果，试试调整任务执行路径。

**适用场景**

任务需求多变或者信息来源多样的应用，如实时新闻摘要、跨领域研究等。

**优势**

- 增强系统的灵活性和适应性；
- 提高任务处理的效率和准确性。

## 图结构 Agentic RAG

利用图数据库表示知识结构，代理通过图遍历和多跳推理，实现更深层次的信息检索和理解。

**架构特点**

- 知识图谱：将实体及其关系表示为图结构，支持复杂查询；
- 多跳推理：代理可以沿图中的路径进行多跳推理，获取简介相关的信息；
- 混合检索：结合向量检索和图检索，提升信息检索的全面性。

**适用场景**

需要理解复杂关系和上下文的应用，如学术研究、法律与合规分析等。

**优势**

- 提升信息检索的深度和广度；
- 增强系统的推理能力和上下文理解能力。

## 文档流程式 Agentic RAG

通过代理自动化处理文档相关任务，实现从文档解析、信息提取到结构化输出的全流程自动化。

**结构特点**

- 📄文档解析：智能体解析文档结构，识别关键信息；
- 🔍信息提取：结合 RAG 技术，从文档中提取相关信息；
- 📊结构化输出：将提取到的信息组织成结构化输出，便于后续流程处理。

**适用场景**

需要处理大量文档的应用，如合同审查、报告生成、表单填写等。

**优势**

1. 显著提高文档处理的效率和准确性；
2. 减少人工干预，降低操作成本。
